# Flujo de trabajo del equipo

Este documento representa un flujo simple y adaptable para trabajar con Git en equipo.

## Flujo recomendado

```txt
Rama -> Cambio -> Pull Request -> Revision -> Integracion
```

## Paso 1: Crear una rama

Antes de modificar la version principal, se crea una rama de trabajo.

Ejemplo conceptual:

```txt
main
 └── mejora-flujo-mi-area
```

La rama permite trabajar sin afectar directamente la version estable.

## Paso 2: Realizar el cambio

El cambio debe tener un objetivo claro.

Ejemplos:

- Actualizar un documento.
- Agregar una politica interna.
- Ajustar un flujo de revision.
- Corregir informacion obsoleta.
- Proponer una mejora.

## Paso 3: Abrir un Pull Request

El Pull Request permite mostrar el cambio al equipo antes de integrarlo.

Debe explicar:

- Que se cambio.
- Por que se cambio.
- Que necesita revision.
- Si hay dudas o riesgos.

## Paso 4: Solicitar revision

Otra persona revisa el cambio.

Puede:

- Aprobar.
- Pedir ajustes.
- Hacer preguntas.
- Sugerir mejoras.

## Paso 5: Integrar el cambio

Cuando el cambio esta aprobado, se integra a `main`.

`main` debe representar la version estable o aprobada.

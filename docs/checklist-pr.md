# Checklist para Pull Requests

Antes de abrir un PR:

```md
[*] Trabaje en una rama, no directo en main.
[*] El cambio tiene un objetivo claro.
[*] El cambio es pequeno y revisable.
[ ] El titulo del PR explica el cambio.
[ ] La descripcion dice que se modifico y por que.
[ ] Indique que necesita revision.
[ ] Mencione dudas, riesgos o impactos si existen.
```

Antes de aprobar un PR:

```md
[ ] Entiendo el objetivo del cambio.
[ ] El cambio corresponde con la descripcion.
[ ] No hay informacion confusa o incompleta.
[ ] No hay conflictos pendientes.
[ ] El cambio puede integrarse sin afectar negativamente al equipo.
```

## Buenas practicas

- Preferir PRs pequenos.
- Usar titulos claros.
- Explicar el contexto.
- Hacer preguntas si algo no se entiende.
- No tomar los comentarios como algo personal.
- No integrar cambios que no fueron revisados.
